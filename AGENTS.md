## 1. ГЛОБАЛЬНЫЕ ПРАВИЛА И ПРОТОКОЛ РАЗРАБОТКИ
1. **OpenAPI — Источник правды:** Проект ведется по принципу API-First. Файл `openapi.yaml` лежит в корне/ресурсах проекта. Любое добавление или изменение эндпоинтов НАЧИНАЕТСЯ с редактирования `openapi.yaml`. Запрещено писать контроллеры, не соответствующие контракту OpenAPI.
2. **TDD (Test-Driven Development):** Каждая фича начинается с тестов.
    * Доменная логика и Application-сервисы покрываются чистыми Юнит-тестами (JUnit 5 + Mockito).
    * Web-слой покрывается интеграционными тестами с использованием `MockMvc` (Spring Boot Test).
3. **Изоляция зависимостей (Чистая архитектура):** Направление зависимостей строго внутрь: `web` -> `application` -> `domain`. Слой `infrastructure` реализует порты из `domain` и `application`.

## 2. АРХИТЕКТУРНЫЕ СЛОИ И ОГРАНИЧЕНИЯ

### 📂 domain
* **Что здесь:** Сущности (Entities/Aggregates: Project, ProjectMember), Объекты-Значения (Value Objects: ProjectId, Rating), Доменные события, Фабрики (ProjectFactory), Билдеры, Кастомные доменные исключения.
* **СТРОГИЙ ЗАПРЕТ:** Запрещены любые импорты из `jakarta.persistence.*` (JPA), `org.springframework.*`, а также любых HTTP/Веб-библиотек. Домен — это чистый POJO.
* **Логика:** Все бизнес-правила и инварианты проверяются ТОЛЬКО здесь (внутри агрегатов или фабрик). Сущности не имеют публичных сеттеров, состояние меняется через бизнес-методы (например, `project.startSprint()`).

### 📂 application
* **Что здесь:** Сценарии использования (Use Cases / Интеракторы), команды и запросы (CQRS), сборщики (Assemblers), транзакционные границы.
* **CQRS Разделение:**
    * **Команды (Мутация состояния):** `CreateProjectCommand`, `JoinTeamCommand`. Обрабатываются через Command Handler'ы. Изменяют состояние через Агрегаты и сохраняют через Порты Репозиториев.
    * **Запросы (Чтение состояния):** `ListProjectsQuery`, `FindProjectByIdQuery`. Обрабатываются через Query Handler'ы. Возвращают готовые read-модели/DTO.
* **СТРОГИЙ ЗАПРЕТ:** Здесь нет бизнес-логики. Здесь нет JPA аннотаций, HTTP-контекста или Web-DTO.
* **Логика Assembler/Orchestrator:** Получает команду, открывает транзакцию (`@Transactional`), достает Агрегат из репозитория (Порта), вызывает метод Агрегата, сохраняет Агрегат обратно в репозиторий.

### 📂 infrastructure
* **Что здесь:** Реализация портов. JPA-сущности (`@Entity`), Spring Data репозитории, адаптеры доменных репозиториев, мапперы (`infra.mapper`: JPA ↔ Domain через MapStruct), интеграции с внешними сервисами, безопасность (`security`).
* **Оптимизация CQRS (Read-слой):** Для обработки Запросов (Queries) разрешено использовать Spring Data проекции или интерфейсы репозиториев, возвращающие DTO напрямую из БД, минуя доменные Агрегаты.
* **СТРОГИЙ ЗАПРЕТ:** Запрещено принимать бизнес-решения. Слой выполняет только техническую работу по доставке/сохранению данных.

### 📂 web
* **Что здесь:** REST-контроллеры, Web-DTO запросов и ответов, мапперы (`web.mapper`: WebDTO ↔ Commands/Queries), валидация входящих данных (`jakarta.validation.*`), глобальная обработка ошибок (`@ControllerAdvice`).
* **СТРОГИЙ ЗАПРЕТ:** Запрещено использовать JPA-сущности или доменные Агрегаты. Контроллеры оперируют только Web-DTO и транслируют их в Команды/Запросы для Application слоя.
* **REST Контракт:** Контроллеры тонкие. Идентификаторы ресурсов передаются только через Path Variables (`/api/v1/projects/{id}`). Списки, детализация, создание и обновление используют РАЗДЕЛЬНЫЕ DTO.

## 4. ПРАВИЛА НАПИСАНИЯ КОДА И ПАТТЕРНЫ

1. **Исключения:** Используй только явные кастомные исключения (например, `ProjectNotFoundException`, `UserAlreadyInProjectException`). Все они должны перехватываться в `web.advice` и отдавать понятный JSON-ответ клиенту с HTTP-статусом (404, 422, 403).
2. **MapStruct:**
    * При маппинге из JPA-сущности в Доменный Агрегат используй фабрики или билдеры домена, так как доменные модели не имеют публичных сеттеров.
3. **JPA Отношения:** Отношения `@ManyToMany` реализуются строго через явную промежуточную join-таблицу. Все перечисления (Enums) сохраняются в БД как строки (`@Enumerated(EnumType.STRING)`).
4. **Транзакции:** Граница транзакции (`@Transactional`) всегда находится на уровне метода Application-сервиса (Command Handler). Обработчики Запросов (`Query Handlers`) выполняются БЕЗ аннотации `@Transactional` (или с `readOnly = true`).

Перед тем как изменять или создавать файлы, убедись, что ты полностью понял, к какому слою относится задача, и сгенерировал соответствующий тест. Сначала покажи мне план изменений.
